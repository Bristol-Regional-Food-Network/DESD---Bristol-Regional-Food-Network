Database Design and Implementation Overview This project uses PostgreSQL as the relational database system and Django ORM to define and manage the schema. The database runs inside a Docker container and communicates with the Django application container through Docker Compose. This setup ensures separation between the application logic and the data layer, while also providing portability and consistency across development environments. The database structure is defined in Django models and applied using migrations. PostgreSQL stores all persistent data, including users, products, orders, and financial records. Database Technology and Libraries The system uses: PostgreSQL 16 (running in Docker) Django ORM psycopg2-binary (PostgreSQL adapter for Python) The required database driver is included in requirements.txt: psycopg2-binary This library allows Django to communicate with PostgreSQL. Without it, the application would not be able to establish a database connection. Environment Configuration Database credentials and connection settings are stored in the .env file. These include: DB_NAME DB_USER DB_PASSWORD DB_HOST DB_PORT The host is set to db, which corresponds to the database service name defined in docker-compose.yml. Docker automatically creates an internal network so that services can communicate using their service names. In settings.py, Django reads these environment variables to configure the database connection using the PostgreSQL backend. Running the Database To build and start the application and database containers, the following command is used: docker compose up --build This performs the following steps: Builds the Django application image. Starts the PostgreSQL container. Waits for the database to become ready. Applies Django migrations. Starts the Django development server. To run in detached mode: docker compose up -d --build Database Schema and Migrations The database schema is defined in: marketplace/models.py When changes are made to models, migrations are generated using: docker compose exec web python manage.py makemigrations Migrations are applied using: docker compose exec web python manage.py migrate Migrations convert model definitions into SQL statements that create or modify tables in PostgreSQL. The initial migration file is located in: marketplace/migrations/0001_initial.py Database Structure The database is structured around several core entities. Producers and Customers represent users of the platform. Producers create Products, and Customers place Orders. Products belong to both a Producer and a Category. Each Product has one Inventory record, which tracks stock levels. Orders are linked to Customers and contain multiple OrderItems. Each OrderItem references a Product and stores the price at the time of purchase to preserve transactional accuracy. Payments are linked one-to-one with Orders to record payment status and provider details. The system also includes financial tracking tables such as PayoutBatch, ProducerPayout, and ProducerPayoutLine, which manage commission calculations and weekly payout processing. Additional supporting tables handle delivery calculations, surplus offers, and content posts. Data Integrity and Constraints The database enforces integrity through: Foreign key relationships One-to-one relationships Unique constraints (e.g., unique email addresses) Check constraints (e.g., non-negative pricing) Indexed fields for performance optimisation These constraints ensure that invalid or inconsistent data cannot be stored at the database level. Accessing the Database To access PostgreSQL directly inside the container: docker compose exec db psql -U django_user -d django_db To list all tables: \dt To describe a table: \d table_name To exit: \q Alternatively, Django’s database shell can be accessed using: docker compose exec web python manage.py dbshell Database Persistence The database uses a Docker volume: postgres_data This ensures that data remains stored even if containers are stopped or restarted. To completely reset the database (including deleting stored data): docker compose down -v docker compose up --build The -v option removes the database volume. Summary The database implementation follows relational design principles and is fully containerised using Docker. Django ORM manages schema definitions and migrations, while PostgreSQL provides reliable data storage and integrity enforcement. This setup ensures portability, structured data management, financial traceability, and separation between application logic and data persistence.
